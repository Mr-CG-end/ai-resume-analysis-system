## 从题目到产品：交付目标

本项目需要在 24 小时内交付一个可公开访问的智能简历分析系统。招聘人员上传单份 PDF 简历后，系统解析并清洗文本，利用 AI 提取候选人信息；招聘人员继续输入岗位描述，系统返回关键词、匹配证据和量化评分。评审人员无需登录即可在线完成整条流程，并能从公开仓库中确认架构、技术选型、部署方法和测试结果。

产品的核心价值不在于建立完整招聘平台，而在于把耗时的人工阅读压缩成一次可解释的分析。评分结果必须能说明“匹配了什么、缺少什么、依据来自哪里”，不能只返回缺乏证据的模型结论。

## 使用者与关键场景

系统面向两类使用者。招聘人员需要快速获得候选人概况和岗位匹配结果；技术评审人员关注功能是否完整、接口是否稳定、异常是否可控，以及代码和部署是否具备基本工程质量。

一次有效使用包含以下过程：

1. 使用者选择一份含可提取文本的 PDF 简历。
2. 前端完成格式和大小预检，随后上传文件。
3. 后端校验文件、提取多页文本、执行清洗并返回结构化档案。
4. 使用者确认解析结果，输入岗位描述。
5. 后端提取岗位关键词，对技能和经历进行匹配，返回分项分数、总分、证据和说明。
6. 使用者可以清空当前结果并分析另一份简历。

## 优先级与交付边界

P0 是公开演示的发布门槛，任一 P0 缺失都视为主流程未完成。P1 对应题目中的加分能力，目标版本应尽量实现，但不得以牺牲 P0 稳定性为代价。

### P0：发布门槛

| 编号 | 需求 | 验收口径 |
| --- | --- | --- |
| FR-PDF-001 | 每次只接受一个 PDF 文件 | 多文件、DOC、DOCX 和图片直接拒绝 |
| FR-PDF-002 | 支持多页原生文本 PDF | 三页测试简历能按页序合并，不遗漏页面 |
| FR-PDF-003 | 同时校验扩展名、MIME 和 `%PDF-` 文件头 | 伪装成 PDF 的文件返回 `415` |
| FR-PDF-004 | 限制文件大小和页数 | 大于 10 MB 返回 `413`，超过 30 页返回 `422` |
| FR-PDF-005 | 清洗提取文本 | 归一空白和换行，移除常见重复页眉页脚，保留段落语义 |
| FR-PDF-006 | 明确不支持无文本 PDF | 扫描件、加密、损坏或无有效文本文件返回可理解的 `422` 错误 |
| FR-AI-001 | 固定返回姓名、电话、邮箱和地址 | 四个键始终存在，未找到时返回 `null` |
| FR-AI-002 | 不得补造候选人信息 | 地址、姓名或日期缺失时保留为空，并返回警告代码 |
| FR-JD-001 | 接收岗位描述文本 | 接受 20 至 10,000 字符，空白或过短内容返回 `400` |
| FR-JD-002 | 提取并规范化关键词 | 去重、忽略大小写差异，保留用于评分的技能和职责词 |
| FR-MATCH-001 | 输出技能匹配结果 | 返回 JD 关键词、匹配关键词和缺失关键词 |
| FR-MATCH-002 | 输出经历相关性 | 返回 0 至 100 的相关性分数和来自简历的证据文本 |
| FR-MATCH-003 | 使用固定公式计算总分 | `0.6 × skill_match + 0.4 × experience_relevance`，按十进制 0.5 向上取整 |
| FR-API-001 | 所有接口返回稳定 JSON | 成功和失败结构与 API 契约一致 |
| FR-API-002 | AI 故障可降级 | 重试一次后退回规则提取或关键词评分，并设置 `degraded=true` |
| FR-UI-001 | 提供完整单页流程 | 在线完成上传、查看档案、输入 JD、查看评分和重新分析 |
| FR-UI-002 | 明确展示状态 | 上传、解析和评分期间显示加载状态，错误信息可操作 |

## 数据与行为约束

`name`、`phone`、`email` 和 `address` 的“必选”含义是接口必须稳定返回这些字段，并且系统必须尝试提取，而不是要求模型在原文缺失时生成答案。地址保留简历中的原始表述，不区分户籍地和现居地；期望薪资也保留原始字符串，不在本版本中处理币种、税前税后或支付周期。

工作年限只根据明确的任职日期区间计算，重叠区间需要去重。日期信息不足时允许 AI 给出辅助判断，但接口仍可返回 `null`，不得根据毕业年份推断工作年限。

两阶段之间通过 `resume_snapshot` 传递状态。快照由第一阶段响应生成，前端仅在当前页面内存中保存；刷新或重新分析时清除。后端不能依赖函数实例内存保存上一次请求，也不能要求 Redis 一定可用。

## 非功能要求

| 编号 | 要求 | 约束 |
| --- | --- | --- |
| NFR-PERF-001 | 同步请求必须受控 | 单次 AI 调用超时 20 秒，最多重试一次 |
| NFR-SEC-001 | 密钥只存在于后端 | AI 和 Redis 凭据使用环境变量，不进入前端或仓库 |
| NFR-SEC-002 | 降低个人信息暴露 | 不记录完整简历、JD、电话、邮箱或地址 |
| NFR-SEC-003 | 原始文件不持久化 | 解析完成后立即释放临时文件，不写入数据库或对象存储 |
| NFR-REL-001 | 可选依赖不影响核心流程 | Redis 连接失败时自动旁路，不返回业务错误 |
| NFR-REL-002 | 输出可追踪 | 每个错误响应包含 `request_id`，日志使用同一标识 |
| NFR-COMPAT-001 | 公开演示可跨域访问 | CORS 仅允许线上前端域名和本地开发地址 |
| NFR-UX-001 | 基本可访问性 | 表单具备可见标签，错误不用颜色作为唯一提示，键盘可完成主流程 |

## 不纳入本次范围

本版本不实现批量上传、DOCX、OCR、用户登录、历史简历库、候选人排行、多岗位对比、向量数据库、RAG、模型微调、异步队列、WebSocket、报告导出、国际化和后台管理。复杂双栏排版和表格只能尽力提取，不承诺完全还原视觉结构。
